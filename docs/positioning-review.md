# HSpace 产品定位评审

**评审日期**:2026-07-04
**评审对象**:HSpace 现行对外定位（README + 落地页）
**评审类型**:定位清晰度 / 差异化真实度 / 卖点可传播性
**结论**:方向正确,清晰度 70 分;需要 1 次收窄 + 1 个商业模式假设,不需要推翻重来。
**决议状态**:2026-07-04 全部决议完毕并已执行第一轮文案迭代,见第 7 节决议列。

---

## 1. 一句话现状

> **私密分享 AI 生成的内容。** 一键把 AI 帮你写好的 HTML 或 Markdown 发布成"链接 + 密码",像递名片一样只交给该看的人。

这是一句**功能描述**,不是**定位**。它回答了"产品做什么",没回答"为谁做、解决他什么场景下的问题、为什么是你"。

---

## 2. 六维评分

| 维度 | 分数 | 诊断 |
|------|------|------|
| 目标用户清晰度 | **2.5 / 5** | "AI 协作者"过宽,没收窄到具体角色;工程师、PM、设计师、咨询师都在范围内,等于没画像 |
| 场景钩子强度 | **4 / 5** | "AI 生成内容"是当下强钩子,且与"私域分发"形成自然张力,有故事可讲 |
| 功能差异化真实度 | **3.5 / 5** | 每人一链 / 版本化 / MCP 入口是真壁垒;密码保护是组合差异,非独家 |
| 卖点可传播性 | **3 / 5** | "私密分享 HTML"不够性感,缺乏场景化包装;用户记不住"为什么我要用它" |
| 商业模式明确度 | **1.5 / 5** | 文档完全没提商业模式,所有功能优先级判断缺锚点——这是定位硬伤 |
| 文案准确性 | **3 / 5** | "私域分发"与"私域流量"混淆;"内容为主 vs 站点为主"用户感知弱 |

**总分:17.5 / 30(约 70 分)**——及格偏上,但还没到"可以放心做增长"的程度。

---

## 3. 真差异 vs 伪差异

这是本次评审最重要的一张表。**真差异是壁垒,伪差异是自我感动。**

| 能力 | 真差异 / 伪差异 | 理由 |
|------|----------------|------|
| 每人一链(独立密码 + 按人统计 + 单独撤销) | ✅ **真差异** | tiiny.host / Netlify / Cloudflare Pages 都没有;解决"发错人怎么办"的真实痛点 |
| 内容版本化(链接不变,历史回滚) | ✅ **真差异** | 竞品做"每次发布 = 新链接",HSpace 做"链接是稳定资产";与 AI 反复迭代天然契合 |
| MCP / OpenAPI 原生入口 | ✅ **真差异(有窗口期)** | 内容诞生在 AI 对话里,分享也在对话里——闭环。但窗口期短,竞品会跟进 |
| 密码保护 | ⚠️ **组合差异** | Netlify Drop 可加密码、Cloudflare Access 可做门;HSpace 的差异是"默认私密 + 边缘验证 + 防爆破"的组合,非单点 |
| "内容为主 vs 站点为主" | ❌ **伪差异** | 用户感知弱;"不发多文件站点"是 Non-goal,不是卖点 |
| "默认私密 vs 默认公开" | ⚠️ **半真差异** | 真实,但"默认私密"本身不是用户要的——用户要的是"可控 + 可撤回",私密只是手段 |

**结论**:真壁垒是**"每人一链 + 版本化 + AI 原生入口"**这三件;密码和私密是把这些壁垒包装起来的外壳,不是壁垒本身。

## 4. 三个核心短板

### 短板 A:目标用户过宽(清晰度 2.5)

"AI 协作者"覆盖太广——用 ChatGPT 写方案的咨询师、用 Cursor 写代码的工程师、用 v0 生成 demo 的设计师,都是"AI 协作者",但他们分享内容的场景、频率、对私密的要求完全不同。

**宽画像 = 没画像**。没画像就没法做文案、没法做渠道、没法判断功能优先级。

### 短板 B:商业模式缺失(明确度 1.5)

文档完全没提商业模式。这导致的连锁反应:

- 路线图里"npm 上架 hspace-mcp、GPT Action 落地"是工程动作,不是商业动作
- "匿名优先 vs 登录能力"的边界没有商业逻辑支撑(为什么要给登录用户更多能力?)
- 滥用防线(频率/体积/熔断)没有"免费 vs 付费"的分层锚点
- 团队无法判断"该往哪个方向投入工程资源"

**没有商业模式的定位,本质是"产品方向"而非"产品定位"。**

### 短板 C:文案术语不精准(准确性 3)

- **"私域分发"**:与营销领域的"私域流量"强混淆,用户第一反应是"这是不是微商工具?"
- **"内容为主 vs 站点为主"**:这是产品内部的架构判断,不是用户语言;用户不觉得"我分享的是内容不是站点"
- **"AI 产物"**:用词略贬义(产物 = 副产品?),建议换"AI 协作成果"或"AI 生成的内容"

## 5. 八条调整建议(按优先级)

### P0 —— 必须改,改了定位才成立

1. **目标用户收窄**:从"AI 协作者"收窄到"用 AI 编程工具协作的开发者"(Cursor / Claude Code / Copilot 用户)。理由:这群人(1)高频生成 HTML/MD demo,(2)有分享给同事/客户 review 的真实场景,(3)对"链接可撤回"敏感,(4)是 MCP/OpenAPI 入口的天然受众。打透这群人,再外扩到 PM/设计师/咨询师。

2. **商业模式定方向**:至少给一个假设。建议起步方向——
   - **个人免费**:匿名 + 登录用户共享,7 天过期、访问量封顶、合集篇数上限
   - **团队订阅($X/月)**:不过期、每人一链无上限、自定义品牌、团队空间
   - **理由**:让"登录能力"有商业锚点,让滥用防线有分层依据,让路线图有取舍依据

### P1 —— 应该改,改了传播效率翻倍

3. **一句话定位拆两层**:
   - **对外**:"AI 协作产物的私密分发站"——给用户/媒体/投资人听
   - **对内**:"HTML/MD 内容的定向分享与撤回控制"——给工程/设计做判断用

4. **"私域分发"换词**:改成"定向分享"或"受限分享",避免与私域流量混淆。"私域"这个词留给商业模式讨论时用。

5. **文案明确张力**:"发布侧匿名,分发侧留痕"——把"匿名优先"和"每人一链可追溯"两个看似矛盾的能力,用一句话讲清楚是"发布者匿名,接收者留痕",解决用户困惑。

### P2 —— 可以改,锦上添花

6. **删掉"内容为主 vs 站点为主"对外文案**:换成"成品内容分享,不是协作编辑"——用户能秒懂。

7. **明说"私密→公开"转化路径**:如果长期要做公开画廊(目前是 Non-goal),要在定位里留口子;如果不做,要明说"我们永远不做公开分享",让用户对产品边界有预期。

8. **域名品牌弱(`zhanjian.space`)**:长期考虑独立短域名(如 `hspace.app` / `hspc.io`)。短期不急,但定位文档里记一笔。

---

## 6. 如果只改一件事

> **把目标用户从"AI 协作者"收窄到"用 AI 编程工具协作的开发者",对外文案围绕该人群场景重写。**

理由:这一件事同时解决两个短板——
- 目标用户清晰度 2.5 → 4
- 卖点可传播性 3 → 4

因为文案一旦聚焦到具体人群,就自动有了场景("Cursor 写完的 demo,一键发链接给客户 review,密码可控、链接可撤回"),用户秒懂为什么是 HSpace 而不是 Netlify Drop。

商业模式可以稍后再定,但**用户画像不收窄,商业模式也定不下来**(不同人群的付费意愿和场景完全不同)。
