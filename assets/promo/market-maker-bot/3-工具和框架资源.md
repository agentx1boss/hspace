# 工具和框架资源

这一篇只回答一个问题:如果我想研究自动挂单, 应该从哪些工具开始?

先立边界:

- 这里没有 StandX 专用 bot。
- 这里不给实盘参数。
- 这里不推荐把 AI 生成的交易代码直接连真钱账户。
- 这里整理的是学习路径和参考框架。

## 一条安全的学习路径

1. 先用只读数据理解 order book。
2. 再写纸面策略, 只输出「如果此刻下单, 应该挂在哪」。
3. 再接模拟环境或小额测试。
4. 再做撤单、断线、异常价差、仓位上限。
5. 最后才考虑实盘。

很多人会跳过第 2 到第 4 步, 直接让 AI 写一个会下单的脚本。那不是快, 是把风控留给运气。

## Hummingbot:从框架理解策略结构

[Hummingbot](https://hummingbot.org/) 是开源的 crypto market making / algo trading 框架。它的价值不只是能跑策略, 更在于你可以看到一个成熟 bot 框架通常怎么拆:

- 数据采集:order book、trades、balances、order status。
- 策略逻辑:根据价格、价差、库存和信号决定下什么单。
- 订单执行:下单、撤单、刷新订单、处理成交。
- 连接器:适配不同 CEX / DEX 的 REST、WebSocket 和节点接口。

[Hummingbot Strategies](https://hummingbot.org/strategies/) 里有一个很清楚的解释:策略就是重复执行数据收集、数据处理和订单执行。V2 里又把组件分成 Executors、Scripts、Controllers, 分别适合一次性任务、学习原型和更长期的模块化部署。

建议阅读顺序:

1. [Strategies](https://hummingbot.org/strategies/):理解策略循环。
2. [Architecture](https://hummingbot.org/strategies/v2-strategies/):理解 Executors、Scripts、Controllers。
3. [Examples](https://hummingbot.org/strategies/v2-strategies/examples/):看 market-making controller 示例。
4. [Strategies V1](https://hummingbot.org/strategies/v1-strategies/):了解 PMM、Avellaneda、perpetual market making 等老策略模板。
5. [Hummingbot API](https://hummingbot.org/hummingbot-api/):理解为什么交易 API 不应该暴露在公网。

## CCXT:理解统一交易 API

[CCXT](https://docs.ccxt.com/) 是一个统一 crypto 交易 API 库, 支持多语言和 100+ 交易所。它适合学习:

- 如何统一读取 ticker、order book、balance。
- 如何抽象 create order / cancel order。
- 不同交易所接口为什么不能简单复制粘贴。
- REST 和 WebSocket 数据流在 bot 里分别承担什么角色。

注意:CCXT 是工具层, 不是策略层。它能帮你和交易所说话, 不会替你解决库存、杠杆、异常行情和收益风险。

## 一个 bot 至少要有的模块

即使只是学习型自动挂单, 也不应该把所有逻辑写成一个 100 行脚本。至少拆成这些模块:

| 模块 | 负责什么 | 新手常见错误 |
|---|---|---|
| Market data | 订阅价格、order book、mark price | 只看 last price, 忽略买卖盘 |
| Quote engine | 算 bid/ask 价格、数量、bps | 没有最小 tick/size 处理 |
| Order manager | 下单、撤单、刷新、查状态 | 以为撤单一定成功 |
| Inventory guard | 控制净仓位、杠杆、最大亏损 | 只管挂单, 不管成交后仓位 |
| Risk switch | 断线、异常波动、价差扩大时暂停 | 没有 kill switch |
| Audit log | 记录每次决策和 API 返回 | 出问题后无法复盘 |

## AI 可以帮什么

AI 很适合帮你做这些事:

- 把官方规则翻成 checklist。
- 解释 bps、post-only、order book。
- 生成只读行情脚本。
- 写 paper trading 原型。
- 帮你审查某段代码有没有硬编码 key、无限重试、裸奔公网 API。

AI 不适合替你决定:

- 该用多少杠杆。
- 挂多近。
- 每边挂多大。
- 什么时候加仓。
- 能不能承受单边成交。

如果 AI 给你一段「复制运行就能赚钱」的代码, 最安全的做法是先把它当成反面教材。

## 推荐资源表

| 资源 | 来源类型 | 适合谁 | 为什么值得读 | 风险提示 |
|---|---|---|---|---|
| [Hummingbot](https://hummingbot.org/) | 开源框架 | 想系统研究 bot 的人 | 看到成熟框架如何处理连接器、策略和执行 | 不代表任何策略可盈利 |
| [Hummingbot Strategies](https://hummingbot.org/strategies/) | 官方文档 | 策略入门 | 把 strategy 拆成数据、处理、执行 | 文档示例不等于实盘配置 |
| [Hummingbot Architecture](https://hummingbot.org/strategies/v2-strategies/) | 官方文档 | 想写原型的人 | 区分 Executor、Script、Controller | 生产部署要更完整风控 |
| [Hummingbot Examples](https://hummingbot.org/strategies/v2-strategies/examples/) | 官方示例 | 想看 market-making 控制器的人 | 有 Dman V3 等 market-making 示例入口 | 示例参数不能照搬到 StandX |
| [CCXT Docs](https://docs.ccxt.com/) | 官方文档 | 开发者 | 统一交易所 API、order book、orders、balances | API 统一不等于撮合规则统一 |
| [Hummingbot API](https://hummingbot.org/hummingbot-api/) | 官方文档 | 想远程控制 bot 的人 | 明确提醒交易 API 是高价值目标, 推荐私有网络 | 暴露公网 API 可能导致未授权交易 |

下一篇是最重要的:风险清单。你能不能研究 bot, 取决于你能不能先解释这些风险。
