# 风险清单:自动挂单不等于自动赚钱

如果只读一篇, 读这篇。

自动挂单最危险的地方, 不是代码难写, 而是它看起来太简单:

> 买一边, 卖一边, 价格走了就撤单重挂。

真正的问题发生在成交、断线、价格跳变、杠杆放大和 API 异常的时候。

## 1. 库存风险

做市不是凭空赚 spread。你的订单会被别人选择性成交。

如果买单成交, 你变成多头。  
如果卖单成交, 你变成空头。  
如果只成交一边, 你就不再是中性的双边挂单者, 而是带着仓位暴露在行情里。

要问自己:

- 净仓位超过多少就暂停?
- 是否需要反向平仓?
- 平仓用 maker 还是 taker?
- 若平仓失败, 最大亏损是多少?

## 2. 单边成交风险

行情单边走时, 你的近端订单可能连续被吃。看起来 bot 一直在提供流动性, 实际上是在不断接盘或不断加空。

如果没有仓位上限和暂停条件, 自动化只会让错误更快发生。

## 3. 杠杆风险

社区案例里有人提到用高杠杆, 但那是个人风险偏好, 不是新手模板。

杠杆会放大:

- 价格跳变。
- 单边成交。
- 未撤订单。
- 保证金不足。
- 止损滑点。

新手应该先用低风险环境理解机制, 不要把「Maker Hours」和「高杠杆」绑在一起。

## 4. 快速行情和滑点

限价单也可能在快速行情里变成 taker 行为。StandX 社区文章里就提到, 不开 ALO 的限价单在快行情中可能立即成交, 带来滑点和费用问题。

所以 bot 必须知道:

- 是否开启 post-only / ALO。
- 订单被拒绝时怎么办。
- 价格离开合格区间时多久刷新。
- 刷新太快是否触发无效行为或 quote stuffing 风险。

## 5. 撤单失败和断线风险

撤单不是本地动作, 是一次外部请求。它可能失败、超时、排队、返回未知状态。

断线时最可怕的不是 bot 停了, 而是:

- 订单还在订单簿上。
- 你不知道它是否成交。
- 你无法及时撤单。
- 你无法更新 TP/SL 或保证金。

所以实盘 bot 至少要有:

- 心跳监控。
- 失败重试上限。
- 启动时同步全部未成交订单。
- 手动 kill switch。
- 异常时只减风险, 不加仓。

## 6. API key 和部署安全

交易 API 是高价值目标。Hummingbot API 文档明确提醒, 这类 API 控制真实交易, 公网暴露、暴力扫描、错误配置都可能导致未授权交易。

最低限度:

- API key 不要写进代码。
- 能只读就先只读。
- 交易 key 尽量限制 IP、权限、额度。
- 不给提现权限。
- 不把 bot 控制接口暴露到公网。
- 不把 `.env` 发给 AI、朋友或截图工具。

## 7. AI 生成代码风险

AI 可以帮你理解机制、写原型、解释代码, 但它也可能:

- 忘记处理撤单失败。
- 把 API key 写进日志。
- 无限循环重试下单。
- 忽略最小下单单位。
- 不处理部分成交。
- 把测试参数当实盘参数。

所以 AI 写的交易代码只能从只读、纸面、模拟开始。没有 code review 和风控清单, 不要连实盘。

## 新手开始前 checklist

下面任意一项做不到, 就先别上真钱:

- 我能解释 maker/taker、post-only/ALO、bps 和 bid/ask spread。
- 我能找到当前 campaign 的合格 band、uptime tier、单边上限和结算规则。
- 我知道 bot 断线后有哪些订单还在场上。
- 我有最大净仓位限制。
- 我有手动停止和撤单方案。
- 我不使用带提现权限的 API key。
- 我不会把 AI 生成代码直接连实盘。
- 我能接受奖励为 0, 甚至交易亏损。

## 一句保命话

自动挂单的第一目标不是赚钱, 是**不要因为自动化而扩大错误**。

当你能清楚解释每个失败场景怎么处理时, 再去研究收益机制才有意义。
